mit6.s081 lab3 - pgtbl

lab pgtbl

这部分实验主要关注页表page table，每个进程都有自己的页表，satp寄存器用于保存当前进程页表的物理地址，页表主要用于将虚拟地址转换为物理地址（实际上是MMU得到要转换的虚拟地址，通过CPU获取页表地址，查询页表得到对应的物理地址）

如果是一个虚拟地址对应页表中的一个表单项，那么64位的系统原则上会有2的64次方个地址，这会使得页表占用很大一部分内存并且难以管理，因此通常是以页面为单位，每个页面对应一个页表表项，因此虚拟地址到物理地址的地址翻译也有所不同：

xv6中虚拟内存地址和物理内存地址结构如下所示：

​

虚拟内存是64位的，其中高25位地址是没有被使用的（因此虚拟内存最多是2的39次方），中间27位是虚拟页号（用作页表索引），最后12位是页内偏移（因为页面大小是4kb，因此offset只能是12位）；物理内存是56位的，前44位是物理页号，剩下12位是页内偏移（与虚拟地址的12位offset完全一致）

xv6中使用三级页表以减少对内存资源的浪费（通常只有一级页表存放在内存中，二级和三级页表根据需求创建），27位虚拟页号分为三个部分，每部分9位，分别用于索引一级页表，二级页表和三级页表；页表大小为4096字节，页表项PTE大小为64比特（也就是8字节），因此一个页面有512个页表项（2的9次方）

因此实际应用中地址翻译的大致流程如下：

  1. 从satp寄存器中取出一级页表的地址
  2. 用虚拟页号的高9位来索引一级页表，得到一个44位的物理页号PPN，加上12位的0便得到了二级页表的基址
  3. 用虚拟页号的中间9位索引二级页表，得到又一个44位的物理页号，同理加上12位的0得到三级页表的基址
  4. 用虚拟页号的低9位索引三级页表，得到44位物理页号，加上虚拟地址低12位的offset便得到了实际的物理地址

最后简单了解下页表项的结构

页表项也是64位的，高10位没有被使用（用作未来扩展），中间44位是物理页号，低10位是标志位，这里主要介绍六个常用的标志位：

Valid：表示页表项是否存在，如果为0，则对页面的引用会导致page fault（即不允许使用）

Readable：表示页面是否可读

Writable：表示页面是否可写

Executable：表示CPU能否执行该页面中的指令

User：表示是否运行用户空间的进程访问该页面，如果为0，则该页面只能在内核中使用

Accessed：表示自上次清除A位以来页面是否被访问过

Speed up system calls

通过在页表中构造一个用户空间和内核空间共享的只读页面usyscall，加速部分系统调用（无需陷入内核，而是直接得到想要的内容）

初始化进程时，为usyscall页面分配空间，这里加速的是getpid系统调用，因此需要将进程id放入usyscall中

添加对usyscall页面的映射（即创建一个虚拟地址->物理地址的页表项），调用mappages函数即可，需要注意创建映射失败时需要移除这之前创建的TRAMPOLINE和TRAPFRAME两个映射

在释放进程空间时记得释放usyscall页面（如果存在的话）

释放进程页表时，移除对usyscall页面的映射

Print a page table

按照指定格式打印当前进程的页表，遍历页表，打印合法的页表项即可，如果页表项指向子页表，则递归调用vmprint函数

Detecting which pages have been accessed

检查哪些页面自上次检查以来被访问过，要求从指定页面开始遍历指定数量的页面，结果通过位掩码形式返回（后续可能会用于页面置换）。具体实现的话通过walk函数遍历页面，对每个页表项检查其A位，如果为1则在位掩码中记录并清除A位（位掩码使用uint64表示，要求一次检查最多遍历64个页面），最后通过copyout函数返回结果即可
