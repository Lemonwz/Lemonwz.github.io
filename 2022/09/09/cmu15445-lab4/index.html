cmu15445 project#4 - Concurrency Control

Project #4: Concurrency Control

这部分实验要求在DBMS中实现锁管理器LockManager以支持查询的并发执行，LockManager以元组作为数据共享的基本单位，事务对元组进行读写操作时需要根据隔离级别从LockManager处获取对应元组的共享锁或排他锁

实验地址：https://15445.courses.cs.cmu.edu/fall2021/project4/

Task #1: Lock Manager

LockManager为当前每个已加锁的数据项（元组）维护一个请求记录的链表，并按请求到达的顺序排序，请求记录记载了提出请求的事务ID，请求锁的模式，请求是否已被授予。LockManager通过一个以数据项标识（元组RID）为索引的散列表来查找对应数据项的链表，这个表称为锁表lock table

  • LockManager类的部分成员如下，lock_table_提供从RID到元组请求队列LockRequestQueue的映射，访问lock_table_前需要先获取互斥锁latch_

  • LockRequestQueue类中，链表request_queue_存放事务对当前元组的锁请求，访问request_queue_前需要先获取互斥锁queue_latch_，尝试获取元组的读写锁失败时，事务线程调用条件变量cv_的wait函数，释放当前持有的queue_latch_，进入等待状态；持有元组读写锁的事务线程执行完毕释放锁时，会调用cv_的notify_all函数唤醒所有等待的事务线程，这些线程会再次尝试获取元组的读写锁；upgrading_记录当前元组中正在升级锁的事务ID

    条件变量condition_variable用于实现线程同步，条件不满足时阻塞线程，满足后唤醒线程

    • 为什么有了互斥锁还要使用条件变量：以当前场景为例，如果使用互斥锁的话，事务线程不满足获取元组锁条件时会释放持有的互斥锁，释放后事务线程不知道什么时候能够满足条件，因此只能不断地轮询 获取互斥锁->获取元组锁->失败释放互斥锁，而在获取元组锁条件真正满足之前（例如某个事务释放了其持有的元组锁），这些轮询操作是没有任何意义的，会导致CPU做无用功及高CPU占用率。这时就需要条件变量，在条件满足之前阻塞事务线程，在条件满足时再唤醒事务线程
    • 条件变量wait函数：对传入的unique_lock解锁，随后将当前线程添加到条件变量的等待线程队列中并阻塞线程（解锁和阻塞这两步是原子的），线程被唤醒后，会重新对unique_lock上锁
    • 条件变量notify_one/notify_all函数：notify_one函数唤醒条件变量等待线程队列中的第一个线程，其余线程继续阻塞；notify_all函数唤醒等待线程队列中的所有线程，其中一个线程会获得锁，其余线程不会继续阻塞，而是以类似于轮询的方式继续尝试获取锁（轮流获取锁）

  • LockRequest类中，txn_id_表示请求事务ID，lock_mode_表示事务请求的锁模式，granted_表示当前请求是否被授权

Transaction类的部分成员如下：

  • state_表示事务当前状态，有GROWING、SHRINKING、COMMITTED和ABORTED四种
  • isolation_level_表示事务当前隔离等级，有READ_UNCOMMITTED、REPEATABLE_READ和READ_COMMITTED三种
  • table_write_set_和index_write_set_存放当前事务对表和索引的写操作记录
  • shared_lock_set_和exclusive_lock_set_存放当前事务持有的共享锁和排他锁对应的RID

事务调用LockShared函数尝试获取S锁，大致流程如下：

  1. 对事务请求合法性检查：

    1.1 如果请求事务状态为ABORTED，返回false

    1.2 如果请求事务隔离级别为READ_UNCOMMITTED，无需获取S锁，设置事务状态为ABORTED，返回false

    1.3 如果请求事务状态为SHRINKING，不能再获取锁，设置事务状态为ABORTED，返回false

    1.4 如果当前事务已持有S锁或X锁，为了避免重复获取锁，返回true

  2. 先拿lock_table_的锁,再拿对应元组请求队列的互斥锁lck,然后再释放lock_table_的锁，顺序不能颠倒

  3. 构造当前事务请求LockRequest插入元组请求队列

  4. 循环判断是否满足获取S锁的条件

    4.1 如果不满足条件，调用条件变量cv_的wait函数阻塞在互斥锁lck上

    4.2 被唤醒后需要检查当前事务是否已经ABORTED，如果是则从请求队列移除事务请求，返回false，否则继续尝试获取S锁

  5. 满足获取S锁的条件，设置对应请求的granted_为true（这部分可有可无，granted_并没有实际用到）

  6. 在事务的shared_lock_set_中添加对应元组RID的记录，返回true

注：正常来说事务因不满足合法性检测或死锁避免被ABORTED后应该抛出对应的异常，在execution_engine中捕获异常并处理，但测试中并没有异常处理部分，而是在返回后检查事务状态是否为ABORTED，并直接调用Abort函数中止事务，因此为了通过测试直接返回false即可

SharedLockRequest函数判断事务是否满足获取S锁的条件，满足返回true，不满足需要等待返回false

  • 调用DeadLockAvoidance函数中止所有优先级较低的冲突事务，返回true表示有事务因死锁避免被中止，这个事务可能持有锁也可能正在等待锁，因此需要调用notify_all函数唤醒所有等待的事务线程检查自己是否被ABORTED
  • 遍历请求队列，如果当前事务之前不存在优先级更高的X锁请求，则授予S锁，否则需要继续等待
  • DeadLockAvoidance函数中ABORTED的请求可能并没有实际从请求队列中移除，如已经授权的锁请求，因此遍历请求队列时需要加一个iter->txn_id_ < req_txn_id的判定条件来过滤掉那些优先级较低的X锁请求，即在当前事务中这些优先级较低的X锁请求已经被视为不存在了

事务调用LockExclusive函数尝试获取X锁，实现流程与获取S锁大同小异

ExclusiveLockRequest函数判断事务是否满足获取X锁的条件，满足返回true，不满足需要等待返回false；实现策略同SharedLockRequest，只是死锁避免时需要中止优先级较低的任何事务，同样遍历请求队列时如果当前请求之前存在任何优先级更高的事务请求，则不能授予X锁

事务调用LockUpgrade函数尝试将其持有的S锁升级为X锁，基本与获取S锁和X锁一致，细节上的不同在与需要通过LockRequestQueue中的upgrading_判断当前元组是否有升级操作正在进行，不能同时有两个升级请求，否则会互相阻塞；其它的注意upgrading_的设置和锁的升级即可

UpgradeRequest函数判断事务是否满足升级锁的条件，策略同获取X锁，即当前事务前不能存在任何优先级更高的锁请求

事务调用Unlock函数尝试释放其持有的所有锁，大致流程如下：

  1. 如果当前事务不持有任何锁，返回false
  2. 先拿lock_table_的锁,再拿对应元组请求队列的互斥锁lck,然后再释放lock_table_的锁，顺序不能颠倒
  3. 如果当前事务正在升级锁，移除当前事务的升级请求
  4. 从元组请求队列中移除当前事务的锁请求（一个事务只会有一条请求记录），并notify_all唤醒所有等待线程
  5. 如果当前事务隔离等级为REPEATABLE_READ且事务状态为GROWING，则事务进入SHRINKING阶段
  6. 从事务的shared_lock_set和exclusive_lock_set中移除对应的RID记录

Task #2: Deadlock Prevention

LockManager采取死锁避免WOUND-WAIT算法实现死锁避免，使用事务ID作为时间戳分配优先级（较老的事务拥有较小的事务ID，从而拥有较高的优先级），如果请求事务的优先级高于持有锁的事务，则中止持有锁的事务并释放锁，否则请求事务等待

实验中要求中止所有优先级较低的冲突事务（对S锁请求来说，持有X锁的事务是冲突事务，对X锁请求来说，任何持有锁的事务都是请求事务），无论这个事务是否被授予锁，设置中止事务状态为ABORTED，返回true

Task #3: Concurrent Query Execution

task3要求支查询的并发执行，顺序扫描拿S锁，插入、更新和删除操作拿X锁

  • 事务隔离级别为RU时不需要获取和释放S锁；事务隔离级别为RC时S锁和X锁在对应读写操作结束后释放（但是不能在读操作结束后释放持有的X锁）；事务隔离级别为RR时S锁和X锁都在事务COMMIT后释放
  • 涉及对索引的修改操作记得向事务的index_write_set中插入对应的记录，插入要记录插入索引项，更新要记录更新前和更新后的索引项，删除要记录删除索引项，以便后续回滚操作。对元组的修改操作不用维护tuple_write_set，在TableHeap对应函数中已经实现了
