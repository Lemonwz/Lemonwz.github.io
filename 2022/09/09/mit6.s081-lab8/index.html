mit6.s081 lab8 - lock

lab lock

这部分实验要求对xv6内存分配器和块缓存进行锁优化，以提高在多核机器上的并行性

什么时候需要加锁？

一个简单而保守的回答是：如果两个进程访问了同一个数据结构，并涉及对这个数据结构的修改操作，那么就需要对这个数据结构加锁

加锁的两种实现方式：

  1. 使用coarse-grain lock，也就是全局锁，在访问某个数据结构的时候，对所有相关数据结构上锁，从而操作系统只能线性运行，失去了并发执行的能力
  2. 拆分锁，将其与数据结构和进程相关联，从而在进行相关操作时调用对应的锁，并且其它进程与之无关的部分不会收到影响（注意任何时刻只有一个进程可以持有锁，锁的获取和释放都是原子的）

死锁（Deadlock）：

死锁产生的条件：

  1. 互斥：请求的资源同一时刻只能被一个进程占有
  2. 占有和等待：进程在请求资源得不到满足而等待时，不会释放已占有的资源
  3. 不可抢占：资源只能由进程自愿释放，不允许其它进程剥夺
  4. 环路等待：进程请求资源的顺序形成了一个环

死锁的解决方法：

  1. 死锁防止：至少破坏四个死锁产生条件中的一个
  2. 死锁避免：对资源分配加以限制，避免死锁的发生
  3. 死锁检测和恢复：检测到死锁时采取措施恢复

Memory allocator

这部分实验要求完成kalloc.c中物理内存分配的锁优化，kmem结构体由一个空闲链表和一个自旋锁组成，分配或释放内存修改空闲链表时要先获取对应的锁，而在多核机器上，一个CPU持有该锁时其它CPU无法进行内存的分配或释放，从而影响了操作系统的性能

因此我们需要为每个CPU维护一个kmem结构体，使得不同CPU上内存的分配和释放可以并行运行（CPU获取各自对应的锁在对应的空闲链表上分配或释放内存）

当一个CPU对应的空闲链表为空，而其它CPU的空闲链表不为空时，这个CPU必须要从其它CPU的空闲链表中“窃取”一部分到自己的空闲链表中

具体实现：

每个CPU对应一个kmem结构体，由自旋锁，空闲链表和用于防止死锁产生的stealist

kinit函数初始化每个CPU对应列表的锁，并将所有空闲内存分配给当前正在运行的CPU

kfree函数不需要太多改动，获取当前CPU id时要关闭中断，释放的物理页面插入当前CPU的空闲链表表头即可（记得获取和释放对应的锁）

kalloc函数需要在原来的基础上添加当前CPU空闲链表为空时从其它CPU空闲链表“窃取”的操作，大致流程如下：

  1. 调用push_off函数关闭中断，获取CPU id，获取空闲链表的锁，如果空闲链表不为空，从表头取出一个空闲页面，释放空闲链表的锁，调用pop_off恢复中断，返回指向这个空闲页面的指针即可
  2. 如果空闲链表为空，则需要“窃取”其它CPU的空闲链表，但我们不能持有当前CPU空闲链表的锁去请求其它CPU空闲链表的锁，否则可能会出现CPU1请求CPU2锁的同时CPU2也在请求CPU1锁的情况，也就是出现了死锁
  3. 因此在请求其它CPU空闲链表的锁之前，要先释放持有的当前CPU空闲链表的锁，从而我们“窃取”得到的页面就不能直接放入当前空闲链表中，因此这里为每个CPU分配了一个stealist用于中转，由于stealist是每个CPU独占的（不存在其它CPU向当前CPU的stealist中写入数据的情况），并且在此之间中断是关闭的（当前进程不会被调度），因此读写stealist是安全的
  4. 这里选择一次性“窃取”64个页面（不是从单个CPU空闲链表中，而是累计64个），遍历不同CPU的空闲链表，获取对应的锁，将“窃取”的空闲页面存放在当前CPU的stealist中，窃取完成后释放对应的锁
  5. 重新获取当前CPU空闲链表的锁，将stealist中“窃取”的空闲页面以头插法存入空闲链表中，后续取出空闲页面返回即可

Buffer cache

这部分实验要求对bio.c中磁盘区块缓存的分配和释放进行锁优化，原本的bcache结构体由块缓存数组，双向空闲链表和一个全局锁组成，由于全局锁的存在，多个进程不能同时访问空闲链表来请求或释放磁盘缓存；并且不同于kalloc中物理页面分配后就独属于该进程，bcache中的区块缓存即使在分配后也是被多个进程所共享的，因此为每个CPU分配其专属块缓存空闲链表的方法是不可行的

实现思路是使用哈希表将blockno通过哈希运算映射到块缓存buf，哈希表中的每个bucket都分配一个锁，请求或释放对应bucket中的块缓存时要先获取锁，这样只有多个进程访问同一个bucket时需要竞争锁，而访问不同bucket互不干扰，从而提升程序的并行度

具体实现：

bcache结构体中定义了哈希表table，将原先包含所有块缓存的链表head拆分为NBUCKET个bucket（通过对NBUCKET取余来将blockno映射到bucket），对应NBUCKET个自旋锁，每个bucket都是一个块缓存链表；此外还包含一个块缓存数组buf以及一个替换锁replacelock（避免重复分配块缓存）

在buf结构体中，增加tick字段表示当前块缓存的最近使用时间，用于LRU替换

binit函数初始化bcache中定义的锁，并将buf中的所有块缓存放入第一个bucket中（refcnt和tick字段的初始值均为0）

brelse函数解除当前进程对指定块缓存的引用，即refcnt减1，如果减1后refcnt为0，说明当前块缓存未被使用，设置其最近使用时间为当前时钟ticks，只在brelse中且refcnt为0时才设置tick字段的原因是块缓存只有在refcnt为0（空闲）时才会被LRU算法检查其最近使用时间，其它情况下tick字段设置了也是没有意义的

bget函数查找由dev和blockno指定的块的缓存，如果存在则直接返回，如果不存在则通过LRU算法找到一个最近最久未使用的空闲缓冲区分配给指定块并返回

实现流程如下：

  1. 通过哈希运算由blockno得到对应的bucket索引，在bucket中查找是否存在指定的块缓存，如果存在，refcnt加1，获取sleep lock返回即可
  2. 如果不存在，则需要遍历整个哈希表，找到最近最少使用的空闲块缓存，类似于kalloc中的实现，由于要获取其它bucket的锁，为了避免死锁产生，我们要先释放持有的当前bucket的锁，在找到空闲块缓存后，在重新获取该bucket的锁
  3. 在开始查找空闲块缓存到找到空闲块缓存的这段时间内，可能有其它进程请求相同dev和blockno的块缓存，也开始在哈希表中查找空闲缓冲区，导致一个磁盘块有两个块缓存，从而造成数据的不一致问题。为了保证绝对的正确性，定义一个全局锁replacelock使得查找+分配空闲块缓存的过程是线性的；在释放当前bucket的锁后立即获取replacelock，在查找+分配完成后释放replacelock
  4. 虽然是在释放当前bucket锁后立即获取的replacelock，但是这期间仍可能有其它进程完成了查找+分配空闲块缓存的操作，因此需要再遍历一次当前bucket检查指定块是否已被缓存（由于持有replacelock，如果不涉及对bucket数据的修改则不需要获取对应的锁）
  5. 再一次查找当前bucket后仍没有找到，则遍历所有bucket（包括当前bucket），通过LRU算法查找最近最少使用的空闲缓存块replace，并且不能释放replace对应bucket的锁（释放之后replace块可能会被其它进程占用）
  6. 如果找到的replace块在当前bucket中（此时已经持有锁），设置缓存块的相关属性，释放bucket的锁和replacelock，恢复中断，返回缓存块
  7. 如果找到的replace块不在当前bucket中，从replace_bucket中取出replace块，释放replace_bucket的锁，获取当前bucket的锁，将replace块放入当前bucket中，随后设置相关属性，释放锁并恢复中断，返回缓存块

修改bpin和bunpin函数，根据buf的blockno映射得到bucket索引，更新引用计数
