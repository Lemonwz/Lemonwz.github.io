mit6.s081 lab5 - cow

lab cow

这部分实验要求实现操作系统中非常常见的一个优化：写时复制（copy-on-write）

为什么需要写时复制？

写时复制可以视作进程间共享页面的一种实现方式，以fork系统调用为例，创建的子进程的初始地址空间是父进程的一个拷贝（但是两个不同的地址空间），因此需要将父进程中的相应页面复制到子进程中，而有时两个进程都仅仅是读数据或子进程创建完成后立即执行exec系统调用（释放当前页面集合，替换为exec相关的页面），此时这些从父进程复制到子进程的页面就失去了其意义；因此我们可以在创建子进程时使其页表指向父进程的页面集合，即共享父进程的物理内存页面，从而减少fork时的复制以提高性能

写时复制：

为了确保进程间的隔离性，所有映射到两个进程的页面都是只读的（通过设置两个进程对应页表项的标志位），在某个时间点，一个进程对共享数据页面执行写操作时会触发page fault，此时系统会为此进程复制该数据页面的一个副本并修改映射（分配一个页面并将数据页面的内容复制到这个页面上），这样两个进程都拥有该数据页面的私有副本，并且都是可读写的，然后重新执行写操作即可

如何识别一个对只读页面的写操作是否需要写时复制？

需要在两个进程页表的页表项中标记写时复制页面，PTE标志位中的最后两位RSW是为内核保留的，这里使用第八位作为COW位，标识该页面是否是一个共享页面（是否需要写时复制）

一些关于page fault的补充知识：

目前在xv6中，一旦进程触发了page fault就会被kill掉，这是一种非常保守的做法；实际上，page fault可以使得虚拟地址到物理地址的映射变得动态起来，内核可以借助page fault动态更新page table，从而实现许多非常有趣的功能

因此，在发生page fault时，我们需要收集一些必要的信息以采取相应的操作：

  1. 引起page fault的虚拟内存地址：当出现page fault时，出错的虚拟地址会被保存在stval寄存器中

  2. 引起page fault的原因：scause寄存器保存了触发usertrap的原因类型

    12: instruction page fault，指令引起的page fault

    13: load page fault，读引起的page fault

    15: store/AMO page fault，写引起的page fault

  3. 发生page fault时的程序计数器：即page fault在用户空间发生的位置

Implement copy-on write

在当前实验之前，xv6中除trampoline page外，一个物理内存页面只属于一个用户进程，也就是说进程退出后可以立即释放相应的物理页面（trampoline page永远不会被释放）；但在该实验中，父进程退出后子进程可能还在使用这些物理页面，因此我们需要对物理页面的引用计数作为能否释放页面的依据

构造一个结构体pgref，由自旋锁（防止并发导致的不一致问题）和引用计数组成，结构体数组pgrefs元素个数为最大物理页面个数，每个物理页面都有其对应的pgref，通过（pa-KERNBASE）/PGSIZE索引

相应的，在kinit函数中初始化每个pgref的锁，在kalloc函数中分配页面时初始化该页面的引用计数为1，在kfree函数中释放页面时检查该页面的引用计数（引用计数等于1才释放页面，大于1时减1即可）

此外还需要一些辅助函数如增加页面的引用计数、减少页面的引用计数以及查询页面当前引用计数；在kref函数中，如果指定物理页面的引用计数为0，说明该页面还没有被分配或没有及时释放，这两种情况都是很严重的错误，因此直接panic；在kderef中，如果指定物理页面引用计数减1后为0，需要直接调用kfree函数将其释放（调用kfree前要先释放锁）

fork系统调用创建完子进程后会调用uvmcopy函数将父进程页表中的内容复制到子进程页表中，xv6中原本的做法是对父进程页表中的每个页面，调用kalloc函数为子进程分配一个新的页面，然后将父进程对应页面的内容复制到子进程页面中，最后建立子进程页表对这个页面的映射

copy-on write的实现省去了分配新页面以及复制页面的过程，修改父进程对应PTE的标志位（设置COW标志位并移除其写标志位），获取该PTE映射的物理内存地址以及修改后的标志位（这样子进程页表中对应PTE也是设置了COW位且只读的），作为参数调用mappages函数将子进程PTE映射到相同的物理内存页面；最后调用kref函数更新该物理页面的引用计数

用户程序试图对一个只读页面进行写操作时，就会触发page fault，因此我们需要在usertrap函数中捕获这一类型的page fault，r_scause函数会返回触发trap的原因类型，这里只需要处理15这一类型的page fault；通过r_stval函数获取出错的虚拟地址，调用相应的处理函数即可

checkcow检查一个虚拟地址对应的PTE是否设置了COW标志位，在检查COW位之前要先验证虚拟地址以及对应的PTE的合法性

确定了触发page fault的页面是COW页面后，调用copyonwrite函数分配新的页面，复制页面内容并修改标志位，大致流程如下：

  1. 通过虚拟地址查询页表获取oldpte和对应的物理内存地址oldpa
  2. 调用kgetref函数获取当前物理页面的引用计数，如果为1（先前调用checkcow已经排除了引用计数为0的情况），则说明没有必要进行copy-on write操作（可能是在为子进程分配页面后父进程对其私有页面执行写操作，此时该页面应该是可读写的，但还没有修改标志位），直接修改PTE的标志位即可
  3. 如果引用计数大于1，调用kalloc函数申请一个新的物理页面newpa
  4. 调用memmove函数将oldpa中的内容移动到newpa中，得到newpte（此时newpte映射到newpa，但标志位是空的）
  5. 设置oldpte的写位并移除其COW位，获取修改后的标志位与newpte做或运算
  6. 调用kderef函数将物理页面的引用计数减1

记录下memcpy和memmove两个函数的区别（在xv6中没有任何区别，memcpy中实际调用的是memmove）：

两个函数都是从src指向的内存地址开始拷贝n个字节到dst指向的内存地址，不同的是memcpy假设两块内存区域没有重叠，而memmove会检测指定内存区域是否重叠从而改变拷贝策略，因此通常memmove的执行速度会比memcpy慢得多

内存区域重叠：

从src拷贝到dst的前两个字节会覆盖src的后两个字节，等拷贝到这两个字节时，就会拷贝错误的数据到dst中

memcpy不会做任何检查，默认从前到后顺序拷贝数据；而memmove会先检查是否存在重叠现象，如果存在则调转拷贝位置，从尾部开始拷贝（或者使用一个临时空间，将src数据拷贝到临时空间，再拷贝到dst中）
